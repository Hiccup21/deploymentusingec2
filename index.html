This project documents the complete process of deploying a personal portfolio website on AWS infrastructure — from spinning up a cloud server to securing it with a custom domain and HTTPS certificate. The goal was to demonstrate real-world cloud engineering skills: server provisioning, web server configuration, DNS management, and SSL/TLS implementation.

This is not a clean, step-by-step tutorial — it's a real deployment log, including every error that showed up and exactly how each one was fixed.

STEP 01
Launch EC2 Instance
Provision a virtual server on AWS

An EC2 (Elastic Compute Cloud) instance was launched using the AWS Management Console. Amazon Linux 2023 was chosen — it's lightweight, well-supported, and optimised for AWS workloads.

STEP 02
Install & Configure Apache
Set up the web server on the instance

Apache HTTP Server (httpd) was installed and started to serve web content on port 80.

bash — EC2 Terminal
# Install Apache
sudo yum install -y httpd

# Start the service
sudo systemctl start httpd

# Enable on boot
sudo systemctl enable httpd

# Verify it's running
sudo systemctl status httpd
❌ Problem

After running systemctl enable httpd, the output showed "preset is disabled" — meaning Apache was not registered to auto-start on reboot.

✅ Fix

Ran sudo systemctl enable httpd explicitly again and confirmed with sudo systemctl status httpd which showed active (running). The preset warning is cosmetic — the service still starts.

A VirtualHost configuration file was also created so Apache knows how to handle requests for the domain:

bash
sudo nano /etc/httpd/conf.d/portfolio.conf
apache — portfolio.conf
<VirtualHost *:80>
    ServerName    emmanuelcloud.space
    ServerAlias   www.emmanuelcloud.space
    DocumentRoot  /var/www/html

    <Directory /var/www/html>
        AllowOverride All
        Require all   granted
    </Directory>

    RewriteEngine on
    RewriteCond %{HTTPS} off
    RewriteRule ^(.*)$ https://%{HTTP_HOST}$1 [R=301,L]
</VirtualHost>

<VirtualHost *:443>
    ServerName            emmanuelcloud.space
    ServerAlias           www.emmanuelcloud.space
    DocumentRoot          /var/www/html
    SSLEngine             on
    SSLCertificateFile    /etc/letsencrypt/live/emmanuelcloud.space/fullchain.pem
    SSLCertificateKeyFile /etc/letsencrypt/live/emmanuelcloud.space/privkey.pem
</VirtualHost>
STEP 03
Deploy Portfolio Files
Upload and serve the website content

The portfolio files were downloaded from GitHub using wget and extracted into Apache's web root directory at /var/www/html/.

bash
# Attempt 1 — failed with 404
wget https://github.com/Hiccup21/Portfolio/archive/refs/heads/main.zip

# Install unzip
sudo yum install -y unzip

# Extract and move to web root
unzip main.zip
sudo mv Portfolio-main/* /var/www/html/

# Fix permissions
sudo chmod 755 /var/www/html/
sudo chmod 644 /var/www/html/index.html
❌ Problem 1 — GitHub 404

Running wget on the ZIP URL returned 404 Not Found. The branch name main in the URL was wrong.

✅ Fix

Opened the GitHub repo in a browser to check the actual branch name, then updated the URL with the correct branch name and re-ran wget.

❌ Problem 2 — 403 Forbidden

After moving files to /var/www/html/, visiting the public IP showed 403 Forbidden. The HTML file was named "Sikiru's portfolio.html" — Apache couldn't find index.html to serve automatically.

✅ Fix

Renamed the file to index.html using sudo mv and deleted the default Apache welcome page at /etc/httpd/conf.d/welcome.conf, then restarted Apache.

bash — rename & fix
# Rename portfolio file to index.html
sudo mv "/var/www/html/Sikiru's portfolio.html" /var/www/html/index.html

# Remove default Apache welcome page
sudo rm -f /etc/httpd/conf.d/welcome.conf

# Restart Apache
sudo systemctl restart httpd
STEP 04
Configure Security Groups
Open the right ports in AWS firewall

AWS Security Groups act as a virtual firewall. The following inbound rules were added to allow web traffic to reach the instance:

TypeProtocolPortSourcePurpose
SSHTCP22My IPTerminal access
HTTPTCP800.0.0.0/0Web traffic + Certbot challenge
HTTPSTCP4430.0.0.0/0Secure web traffic
❌ Problem — ERR_CONNECTION_REFUSED

Even after opening port 80, the browser showed ERR_CONNECTION_REFUSED. Checking with sudo ss -tlnp | grep :80 returned nothing — Apache wasn't binding to port 80 at all.

✅ Fix

The VirtualHost config file portfolio.conf was missing. Once it was created with the correct <VirtualHost *:80> block and Apache restarted, port 80 started listening correctly.

⚠️Important: Port 80 must stay open even after HTTPS is enabled — Let's Encrypt uses an HTTP challenge on port 80 to verify domain ownership during certificate issuance.
STEP 05
Custom Domain via Namecheap
Pointing emmanuelcloud.space to the EC2 instance

An Elastic IP was allocated and associated to the EC2 instance to ensure the public IP remains the same even after reboots. DNS A records were then added in Namecheap's Advanced DNS panel to point the domain to this IP.

bash — verify public IP
curl ifconfig.me
# Returns: 3.213.211.14
TypeHostValueTTL
A Record@3.213.211.14Automatic
A Recordwww3.213.211.14Automatic
❌ Problem 1 — Couldn't delete Namecheap records

Some default Namecheap records (parking page, CNAME for www) could not be deleted directly. The delete button was greyed out for certain record types.

✅ Fix

Instead of deleting, used the edit (pencil) icon to overwrite existing records with the new A record values. For the CNAME www conflict, changed its type to A Record first, then set the Elastic IP value.

❌ Problem 2 — Wrong IP in DNS

Certbot's verification failed with a connection timeout on IP 3.213.211.14 but the Namecheap A records were still pointing to the old default Namecheap IP, not the Elastic IP.

✅ Fix

Ran curl ifconfig.me on the EC2 instance to confirm the actual Elastic IP (3.213.211.14), then updated both A records in Namecheap to this exact IP and waited for DNS to propagate.

bash — verify DNS propagation
nslookup emmanuelcloud.space

# Expected output:
# Name:    emmanuelcloud.space
# Address: 3.213.211.14  ✅
ℹ️DNS changes can take up to 24 hours to propagate globally. Use dnschecker.org to monitor propagation across multiple regions in real time.
STEP 06
SSL Certificate with Let's Encrypt
Securing the site with free HTTPS

Certbot was installed to obtain a free SSL/TLS certificate from Let's Encrypt and configure Apache to serve traffic securely over HTTPS.

bash — install Certbot
sudo yum install -y epel-release
sudo yum install -y certbot python3-certbot-apache
bash — request certificate
sudo certbot --apache \
  -d emmanuelcloud.space \
  -d www.emmanuelcloud.space
❌ Problem 1 — Certbot timeout

Certbot failed with: "Fetching http://emmanuelcloud.space/.well-known/acme-challenge/…: Timeout during connect (likely firewall problem)". The Elastic IP wasn't properly associated to the instance at the time.

✅ Fix

Went to EC2 → Elastic IPs, confirmed the IP was allocated, then used Actions → Associate to link it to the correct instance. Updated Namecheap DNS to match the new IP, waited for propagation, then re-ran Certbot.

❌ Problem 2 — No VirtualHost on port 80

Certbot returned: "Unable to find a virtual host listening on port 80 which is currently needed for Certbot to prove to the CA that you control your domain."

✅ Fix

Created /etc/httpd/conf.d/portfolio.conf with a proper <VirtualHost *:80> block including ServerName and ServerAlias, restarted Apache, then re-ran Certbot successfully.

❌ Problem 3 — VirtualHost conflict

During Certbot's auto-install step, it asked to select a VirtualHost and returned: "The selected vhost would conflict with other HTTPS VirtualHosts within Apache. Could not install certificate."

✅ Fix

Manually edited portfolio.conf to include both the port 80 and port 443 VirtualHost blocks with the SSL certificate paths hardcoded. Restarted Apache — HTTPS worked perfectly.

bash — auto-renewal check
# Test renewal works correctly
sudo certbot renew --dry-run

# Certbot sets up auto-renewal via systemd timer
# Certificate expires: 2026-09-08 — renews automatically
✅SSL certificate successfully issued and deployed. All HTTP traffic now automatically redirects to HTTPS via Apache's RewriteRule. The 🔒 padlock appears in the browser.
STEP 07
Full Troubleshooting Log
Every error hit during this deployment and how each was resolved

The following is a complete record of all issues encountered, the exact error messages received, the diagnostic steps taken, and the final fixes applied.

🔴 Error #1 — GitHub ZIP returned 404 Not Found
terminal output
--2026-06-07 05:29:13--  https://github.com/Hiccup21/Portfolio/archive/refs/heads/main.zip
Resolving github.com... 140.82.113.3
Connecting to github.com|140.82.113.3|:443... connected.
HTTP request sent, awaiting response... 404 Not Found
ERROR 404: Not Found.

Cause: The branch was not named main in the GitHub repo — the URL was incorrect.

Fix: Opened the GitHub repo in a browser, identified the correct branch name, updated the wget URL, and re-ran the command successfully.

🔴 Error #2 — 403 Forbidden on public IP

Cause: The portfolio HTML file was named "Sikiru's portfolio.html" — Apache couldn't serve it as the default page because it looks for index.html. Also, the default Apache welcome page at /etc/httpd/conf.d/welcome.conf was blocking custom content.

Fix: Renamed the file to index.html using sudo mv, removed welcome.conf, and restarted Apache.

🔴 Error #3 — ERR_CONNECTION_REFUSED in browser

Cause: Apache was active but wasn't binding to port 80 because no VirtualHost config existed. Running sudo ss -tlnp | grep :80 returned nothing.

Fix: Created /etc/httpd/conf.d/portfolio.conf with the correct VirtualHost block. After restarting Apache, curl http://localhost returned the portfolio HTML. The browser then loaded the site successfully.

🔴 Error #4 — Certbot domain verification timeout
certbot error
Domain: emmanuelcloud.space
Type:   connection
Detail: 3.213.211.14: Fetching http://emmanuelcloud.space/.well-known/
acme-challenge/R9h_Q4JE...: Timeout during connect (likely firewall problem)

Cause: The Elastic IP (3.213.211.14) was allocated but not correctly associated to the EC2 instance. Running curl ifconfig.me returned nothing — the instance had no reachable public IP.

Fix: Went to EC2 → Elastic IPs → Actions → Associate Elastic IP address, selected the instance, confirmed the association. Updated the Namecheap A records to 3.213.211.14. Verified with nslookup emmanuelcloud.space which returned the correct IP, then re-ran Certbot.

🔴 Error #5 — Certbot VirtualHost conflict, certificate not installed
certbot error
Successfully received certificate.
Certificate is saved at: /etc/letsencrypt/live/emmanuelcloud.space/fullchain.pem

We were unable to find a vhost with a ServerName or Address of www.emmanuelcloud.space
The selected vhost would conflict with other HTTPS VirtualHosts.
Could not install certificate

Cause: Certbot saved the certificate successfully but couldn't auto-configure Apache because of conflicting VirtualHost definitions — the ssl.conf and portfolio-le-ssl.conf files clashed.

Fix: Manually edited /etc/httpd/conf.d/portfolio.conf to include both the port 80 redirect block and the full port 443 SSL block with the certificate paths pointing to /etc/letsencrypt/live/emmanuelcloud.space/. Restarted Apache — HTTPS worked immediately.

✅After resolving all 5 errors, the site loaded correctly at https://emmanuelcloud.space with a valid SSL padlock.
